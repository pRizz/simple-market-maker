<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin request to server service | Raw settings and API key input enters trusted server code. |
| server service to database | Plaintext provider keys must be encrypted before crossing into persistence. |
| server service to browser/API caller | Only safe metadata may cross back toward client code. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | `provider-key-encryption.ts` | mitigate | Use AES-256-GCM with random IV and auth tag, never store plaintext, and never return the encryption secret. |
| T-02-02 | Information Disclosure | Provider-key CRUD responses | mitigate | Map rows to `ProviderApiKeyMetadata`; tests assert plaintext, ciphertext, IV, and auth tag are absent from returned objects. |
| T-02-03 | Tampering | Missing encryption secret fallback | mitigate | Saved-key create, update, and decrypt fail closed when `PROVIDER_KEYS_ENCRYPTION_SECRET` is missing or too short. |
| T-02-04 | Elevation of Privilege | Disabled saved key | mitigate | `setProviderKeyEnabled` persists enabled state for Plan 03 resolver checks; disabled keys remain stored but are not eligible for persisted credential use. |
| T-02-05 | Repudiation | Key metadata updates | mitigate | Repository updates `updated_at`, `validation_status`, and `last_validated_at` metadata so admin-visible state reflects the last server action without exposing the secret. |
| T-02-06 | Information Disclosure | Raw provider URL/key in validation errors | transfer | Plan 03 owns provider validation and sanitizer behavior; Plan 02 stores only sanitized validation metadata fields. |
| T-02-07 | Tampering | Sample source bypassing server-side gate | transfer | Plan 04 owns server-side `showSampleData` enforcement; Plan 02 exposes settings persistence needed for that gate. |
| T-02-08 | Spoofing | Public deployment/no-auth risk | accept | Single-admin/no-auth is a locked product constraint; Plan 05 documents the deployment risk. |
| T-02-09 | Tampering | Unsupported provider key CRUD | mitigate | Provider-key service checks Plan 01 key-management capability helpers before encryption or repository mutation, so `sample` and `twelve_data` cannot create/update/disable/delete saved keys in Phase 1. |
</threat_model>
